# 面试题库系统 — 数据打标策略设计

针对你提出的问题：“是否应该在这个阶段打标签？我最终要利用 AI 来完成。”

这是一个非常关键的设计决策。核心原则是：**好钢（高级 AI 计算）要用在刀刃上。**

## 打标策略：两级分层打标 (Two-Tier Tagging)

我们不能完全不在第一阶段打标，也不能把所有细粒度的标签都塞在第一阶段。

### 第一层 (Phase 1)：**粗粒度领域标签 (Broad Domain Tags)**
**执行时机**：由 Agent (LLM) 在读取原始面经提取 JSON 时顺手完成。
**目的**：初步筛除或分类明显不是目标方向的题，便于后续聚合。
**能打什么标？**
- `type`: "算法" / "八股" / "场景题" / "HR行为面"
- `difficulty` (整体体感): "Hard" / "Medium" (从原文字里行间体会)
- *注意：不要在这里打诸如 "MySQL索引优化"、"AQS底层" 这种极细粒度的知识点标签。*

### 第二层 (Phase 2)：**细粒度知识图谱打标 (Fine-Grained Concept Tags)**
**执行时机**：**在 Python 脚本完成几千道题的去重之后**。此时我们得到了一个干净的、按频次倒序的 `question_bank.json` (比如只剩下 300 道不重复的核心题)。
**目的**：为最终的 300 道题打上精确的知识树标签，用于 "场景 C：按知识点专项突破" 的检索。
**由谁来做？**：利用高级 AI (如 Gemini 2.5 Pro)，遍历这去重后的 300 道干净题目，大批量生成精准标签。

---

## 为什么不在 Phase 1 阶段让 AI 把细粒度标签全打了？

这也是为什么我想把你的高级 Schema 拆分到第二阶段的根本原因。这是一个工程上的 Trade-off：

1. **AI 幻觉与对齐问题**：
   面经 A 里的原文是：“问了 HashMap 结构”。AI 在 Phase 1 可能打上标签：`["Java基础", "集合框架"]`。
   面经 B 里的原文是：“HashMap 1.8 为什么要红黑树？” AI 在 Phase 1 可能打上标签：`["JDK特性", "数据结构"]`。
   **结果**：后续 Python 脚本想把它们聚类到一起时发现标签不一样，增加处理难度。

2. **计算冗余 (Token Waste)**：
   "什么是 CAS？" 这道题在你的 1300 篇面经里可能出现了 200 次。
   如果让 AI 在 Phase 1 就打细粒度标签，AI 相当于浪费了 200 次算力在同一道题上反复打 `["并发编程", "原子操作"]` 标签。

3. **上下文缺失的单点误判**：
   在一篇很短的面经里单独看一道题，AI 很难打准深度标签。而且原始面经里经常有错别字（比如把 "Spring AOP" 写成 "ApO"），这会干扰 AI 第一时间的判断。

## 结论与行动建议

你最终确实是用 AI 来完成一切，但**AI 的接入时机决定了系统的效率和一致性**。

**推荐做法**：

1. **Phase 1 (现在)**：依然修改提取 Prompt，只让 AI 打最基础的分类（如 "算法"、"八股"、"项目深挖"）。
    ```json
    { "type": "算法", "question": "反转链表" }
    ```
2. **Phase 2 (去重后聚类)**：去重后的干净题库，专门开一个 **Tagging Agent WorkFlow**，让大模型遍历这去重后的终极列表。
    ```json
    // AI 针对这唯一一道干净的题目，生成最权威的标签集
    {
       "question": "HashMap 为什么不能并发？",
       "tags": ["Java并发", "集合底层", "故障排查"],   // <--- 这就是你要的终极标签
       "core_keywords": ["..."]
    }
    ```

**总结**：Phase 1 做**过滤和骨架**，Phase 2 (聚合后) 用 AI 做**点睛和深加工**。

如果你觉得这个策略合理，并且不浪费你大模型的算力和额度，那么 `json_schema.md` 目前的极简（只提供分类框架）设计就是合适的，我们就准备进入写脚本开发阶段了。
