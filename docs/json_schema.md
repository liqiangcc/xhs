# Phase 1: 极简流面经结构化提取 Schema (Data Structuring)

基于你的建议，我们采用极度扁平、极简的 JSON 结构作为**第一阶段（数据提纯）**的唯一输出标准。

这种设计的优势：
1. **超级省 Token 且速度快**：抛弃了对象数组和细分类型，直接吐字符串数组。
2. **便于 Phase 2 聚合算法处理**：Python 脚本读取这个列表做相似度合并时，不需要解析深层结构，直接操作字符串就行。

---

## 最终 JSON Schema (大模型直接输出协议)

在大模型批量扫描 `note_desc/*.txt` 时，必须强校验该结构并输出：

```json
{
  "note_id": "66d3b9a0000000001a030000",
  "source": "小红书",
  "company": "百度",
  "position": "Java后端",
  "round": "未注明",
  "level": "未知",
  "year": 2025,
  "date": "未知",
  "questions": [
    "算法：反转链表",
    "HashMap 为什么不能并发？",
    "死锁了如何排查？"
  ]
}
```

---

## 给大模型（提取 Agent）的规则要点：

1. **`note_id`**：直接等于当前解析的文本文件名，前面拼上平台来源。
2. **`company`, `position`, `round`**：基于原文语境推演，找不到确切词汇一律填 `"未注明"` 或 `"未知"`，绝对禁止大模型捏造。
3. **`year`, `date`**：
   - 如果原文有“春招倒计时”、“3月15日面”等字眼，尝试推断。如果没有，填 `"未知"`。
4. **`questions` 重点规则**：
   - 提取原文中真实被问到的知识点和题目。
   - 保留原滋原味的表述（不要帮作者改写成完美的话术，以免丢失这篇面经的特性）。
   - 【重要】如果是原作者写的答案、吐槽或引申的碎碎念，**坚决不要**放进 `questions` 数组！
   - 对于明显的算法题或场景题，可以在前面自己加上 `算法：` 或 `场景：` 的前缀（正如你提议的 `"算法：反转链表"`），方便后面分类。
