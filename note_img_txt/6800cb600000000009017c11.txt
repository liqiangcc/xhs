银行行业Java后端面试题必问10道

1. 银行系统中如何设计高并发下的账户余额更新?
问题: 在银行系统中，当多个用户同时操作同一个账户时，如何保证账户余额的一致性?
回答: 高并发下更新账户余额要用到分布式锁或数据库事务隔离。比如用 Redis 分布式锁来控制同一时间只能一个人修改账户余额。或者用数据库的悲观锁 (select ... for update) 来确保修改时不会冲突。此外，还可以用消息队列异步处理扣款和到账请求，保证顺序正确。

2. 如何理解 Spring 的 AOP 和它在银行系统中的应用?
问题: AOP 是什么? 在银行系统中有哪些应用场景?
回答: AOP 就是面向切面编程，简单说就是把一些通用功能 (比如日志记录、权限校验) 从主业务逻辑里抽取出来，统一管理。在银行系统里，可以用 AOP 来记录交易日志、检查用户的权限、监控性能等。这样代码更清晰，也 不用在每个地方都写重复的逻辑。

3. 银行系统中如何防止 SQL 注入攻击?
问题: SQL 注入是什么? 如何在银行系统中防止它?
回答: SQL 注入是黑客通过输入恶意的 SQL 语句来攻击数据库。防止 SQL 注入最简单的方法就是使用预编译语句 (PreparedStatement) , 不要直接拼接 SQL 字符串。此外，还 要对用户输入的数据进行严格校验，比如只允许数字或者特定格式的 字符串。

4. 在银行系统中如何设计一个安全的支付接口?
问题: 支付接口的安全性如何保障?
回答: 支付接口需要多层防护。首先是数据加密，用 HTTPS 协议传 输数据，确保通信过程不被窃听。其次是签名验证，每次请求都要带 上签名信息，服务器验证签名是否正确。最后还有防重放攻击，可以 通过加入时间戳和唯一标识符来实现。

5. 银行系统的缓存策略是什么?
问题: 银行系统中常用哪些缓存策略?
回答: 银行系统里常用的缓存有 Redis 和 Memcached。比如用户 的基本信息、账户余额这些经常查询的数据可以放在 Redis 里，减 少对数据库的压力。另外，为了防止缓存击穿 (很多人 多) ，可以用热点数据预热或 更多 保护 银行行业Java后端面试题必问合集.docx

6. 如何设计银行系统的事务管理?
问题: 银行系统中的事务管理有哪些要点?
回答: 银行系统里的事务非常重要，比如转账时要保证钱从一个账户 扣掉的同时，另一个账户能正确增加。可以用数据库的 ACID 特性 来实现，也就是原子性、一致性、隔离性与持久性。如果涉及多个服 务，可以用分布式事务解决方案，比如 TCC (Try-Confirm-Cancel) 模式。

7. 银行系统中的定时任务如何设计?
问题: 银行系统中有哪些常见的定时任务场景?
回答: 银行系统里有很多定时任务，比如每天凌晨生成报表、清理过 期数据、给用户发送账单通知等。可以用 Quartz 或者 Spring Task 来实现定时任务，设置好执行时间 和频率就行。不过要注意并发问题， 比如同一个任务不能同时跑多次。

8. 如何设计银行系统的日志系统?
问题: 银行系统的日志系统有哪些特殊要求?
回答: 银行系统对日志的要求很高，必须能完整记录每笔交易的过程。 可以用 ELK (Elasticsearch, Logstash, Kibana) 来收集和分析日志， 确保日志能长期保存并且方便查询。另外，日志里不能包含敏感信息， 比如用户的密码或者完整的卡号。

9. 银行系统中的分布式限流如何实现?
问题: 如何限制银行系统的访问量，防止系统崩溃?
回答: 可以用分布式限流工具，比如 Sentinel 或者 Hystrix。原理 是设置一个阈值，当单位时间内请求数超过这个值就拒绝后续请求。 比如每秒只能处理 1000 个请求，超过的部分会返回错误提示。这 样既能保护系统稳定运行，又能保证用户体验。

10. 银行系统的负载均衡策略有哪些?
问题: 负载均衡在银行系统中有哪些常用策略?
回答: 负载均衡就是把请求分发到不同的服务器上，让它们一起工作。 常见的策略有轮询法 (按顺序分配请求) 、最少连接法 (优先分配给 空闲的服务器) 和 IP 哈希法 (根据用户 IP 固定分配到某台服务器) 。 银行系统一般用硬件设备 (比如 F5) 或者软件 (比如 Nginx) 来做 负载均衡，确保高可用性和高性能。
