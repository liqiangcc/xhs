# tcp和udp的区别

## 核心结论

TCP 是面向连接、可靠、有序、面向字节流的传输协议；UDP 是无连接、尽力而为、面向报文的传输协议。选择时主要看是否需要可靠性、低延迟、顺序保证和应用层控制能力。

## 1 分钟版

TCP 通信前需要三次握手建立连接，传输中通过序号、确认、重传、滑动窗口、拥塞控制保证可靠和有序，适合文件传输、HTTP、数据库连接等。UDP 不建立连接，不保证可靠、不保证顺序，但头部小、延迟低，适合直播、语音、游戏、DNS 等对实时性或应用层自定义可靠性要求更高的场景。

## 3 分钟版

TCP 把数据看成字节流，所以应用层需要处理粘包和拆包。UDP 保留报文边界，一次 send 对应一个 datagram，但报文过大可能分片，丢包风险增加。TCP 的可靠性来自确认应答和超时重传，有序性来自序列号，流控通过接收窗口避免压垮接收方，拥塞控制通过慢启动、拥塞避免等机制保护网络。UDP 简化这些机制，把更多控制权交给应用层。

## 关键细节

- TCP 面向连接，UDP 无连接。
- TCP 可靠有序，UDP 不保证可靠和顺序。
- TCP 是字节流，UDP 是报文。
- TCP 头部和状态维护成本更高。

## 原理机制

- TCP 三次握手建立双方初始序列号。
- ACK 和重传保证可靠投递。
- 滑动窗口实现流量控制。
- UDP 只提供端口复用和基本校验，传输控制由应用决定。

## 项目经验版

后端常见业务接口、RPC 和数据库访问通常基于 TCP，因为更需要可靠和有序。实时音视频、埋点上报、游戏状态同步可能选择 UDP 或基于 UDP 的协议，再在应用层做丢包处理、重传或冗余包。

## 常见追问

- TCP 为什么要三次握手？
- TCP 粘包是什么原因？
- UDP 一定不可靠吗？
- QUIC 和 UDP 是什么关系？

## 易错点

- 不要说 UDP 没有任何校验。
- 不要把 TCP 说成消息协议，它是字节流。
- 不要忽略应用场景对可靠性和延迟的权衡。
