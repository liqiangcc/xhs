# Redis为什么快

## 核心结论

Redis 快主要因为数据在内存中、采用高效数据结构、单线程命令执行避免锁竞争、基于 IO 多路复用处理大量连接，并且协议和实现都比较轻量。

## 1 分钟版

Redis 的核心读写在内存中完成，避免了磁盘随机 IO。它的数据结构针对不同场景做了优化，例如 SDS、ziplist/listpack、skiplist、hashtable 等。命令执行主线程单线程化，避免了多线程锁竞争和上下文切换。网络层使用 epoll 等 IO 多路复用，一个线程可以管理大量连接。Redis 6 之后引入多线程主要用于网络 IO 读写，命令执行仍保持单线程模型。

## 3 分钟版

可以从存储、计算、网络三个层面解释。存储上，Redis 是内存数据库，访问延迟远低于磁盘。数据结构上，Redis 为字符串、哈希、列表、集合、有序集合设计了紧凑或高效编码，减少内存和 CPU 开销。执行模型上，单线程按事件循环处理命令，避免共享数据加锁；命令本身通常很短，所以单线程可以支撑高吞吐。网络上，IO 多路复用让 Redis 不需要为每个连接创建线程。需要注意的是，如果执行大 key、慢 Lua 或复杂命令，仍然会阻塞主线程。

## 关键细节

- 快不等于所有命令都快，大 key 和 O(n) 命令会拖慢。
- 单线程指命令执行主流程，不代表 Redis 没有后台线程。
- Redis 6 多线程主要优化网络 IO。
- 持久化和复制会带来额外开销。

## 原理机制

- Reactor 事件循环负责连接读写和命令分发。
- 内存访问延迟远低于磁盘访问。
- 单线程执行天然避免共享数据结构锁竞争。
- 高效编码减少内存占用和 CPU cache miss。

## 项目经验版

项目里用 Redis 做热点缓存时，要控制 key 大小和 value 序列化成本。对于排行榜用 ZSet，对于计数用 String/incr，对于对象缓存用 Hash 或 JSON 字符串。线上排查 Redis 慢时会看 slowlog、bigkeys、命令耗时、网络延迟和持久化压力。

## 常见追问

- Redis 单线程为什么还能高并发？
- Redis 6 多线程解决什么问题？
- big key 会造成什么影响？
- Redis 和本地缓存怎么选？

## 易错点

- 不要说 Redis 快只因为内存。
- 不要误以为 Redis 所有模块都是单线程。
- 不要忽略慢命令对主线程的阻塞。
