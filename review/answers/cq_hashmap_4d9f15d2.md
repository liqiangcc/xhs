# HashMap原理

## 核心结论

HashMap 是基于数组加链表或红黑树实现的散列表，通过 key 的 hash 定位桶位，再在桶内比较 key 找到元素。它关注的是平均 O(1) 的查找和写入，但需要理解 hash 扰动、扩容、链表树化以及线程不安全问题。

## 1 分钟版

put 时先计算 key 的 hash，使用数组长度减一与 hash 做位运算定位桶。如果桶为空直接放入；如果桶里已有节点，就按 hash 和 equals 判断是覆盖还是追加。JDK 8 之后桶内冲突先用链表，链表过长且数组容量足够时转为红黑树。元素数量超过阈值后触发扩容，容量通常翻倍，节点会重新分布到新表。

## 3 分钟版

HashMap 的数组长度保持为 2 的幂，这样可以用 `(n - 1) & hash` 快速取模。hash 扰动会把高位信息混入低位，减少低位分布差导致的冲突。扩容由 `capacity * loadFactor` 控制，默认负载因子 0.75，是空间和冲突概率之间的折中。JDK 8 的扩容迁移可以根据旧容量对应的 bit 判断节点留在原位置还是移动到 `oldIndex + oldCapacity`。链表树化需要同时满足链表长度阈值和最小容量阈值，避免小表过早树化。

## 关键细节

- 默认初始容量是 16，默认负载因子是 0.75。
- key 可以为 null，null key 通常落在 0 号桶。
- 树化阈值通常是链表长度大于等于 8，并且表容量至少 64。
- HashMap 不是线程安全集合，并发写可能导致数据覆盖、丢失或结构异常。

## 原理机制

- 数组负责快速定位桶。
- hashCode 和 equals 共同决定 key 是否相同。
- 链表和红黑树负责处理 hash 冲突。
- 扩容会降低冲突，但会带来一次性迁移成本。

## 项目经验版

项目里如果能预估元素数量，会在创建 HashMap 时指定合适容量，减少扩容成本。自定义对象作为 key 时一定要同时重写 hashCode 和 equals，并保证参与计算的字段尽量不可变。多线程写入场景不要用 HashMap，优先使用 ConcurrentHashMap 或在外层做同步控制。

## 常见追问

- 为什么 HashMap 容量要是 2 的幂？
- JDK 7 和 JDK 8 的 HashMap 有什么差异？
- 为什么重写 equals 必须重写 hashCode？
- HashMap 为什么线程不安全？

## 易错点

- 不要把 HashMap 说成完全有序。
- 不要只说数组加链表，要补充 JDK 8 的红黑树。
- 不要忽略扩容和负载因子的作用。
