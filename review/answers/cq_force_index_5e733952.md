# 如果查询优化器选错了索引怎么办？

## 核心结论

优化器选错索引时，先用 EXPLAIN、执行耗时和数据分布确认原因，再优先通过更新统计信息、改写 SQL、调整索引解决。force index 可以作为短期兜底，但不应成为长期唯一方案。

## 1 分钟版

排查时先看 EXPLAIN 的 type、key、rows、filtered、Extra，确认实际使用了哪个索引。常见原因包括统计信息过旧、字段区分度低、条件组合不符合最左前缀、排序分页影响选择、返回行数估算错误。处理方式可以是 ANALYZE TABLE 更新统计信息，补充或调整联合索引，改写 SQL 让条件更可选择，必要时使用 FORCE INDEX 或 USE INDEX 临时指定索引。

## 3 分钟版

MySQL 优化器会基于成本估算选择执行计划，成本依赖统计信息、索引基数、扫描行数、回表成本、排序成本等。如果统计信息不准，或者数据倾斜严重，优化器可能选择看似便宜但实际很慢的索引。处理顺序建议是先复现慢 SQL，拿到实际参数和执行计划；再比较不同索引下的执行时间；然后从数据分布、索引设计和 SQL 写法上修复。索引提示适合紧急止血，但数据规模变化后可能反过来变慢，所以需要监控和定期复查。

## 关键细节

- ANALYZE TABLE 可以刷新统计信息。
- FORCE INDEX 强制使用指定索引，USE INDEX 是建议使用。
- 联合索引顺序要结合等值、范围、排序和覆盖查询设计。
- 数据倾斜时平均统计信息可能误导优化器。

## 原理机制

- 优化器选择的是估算成本最低的执行计划。
- 索引基数和过滤率影响扫描行数估算。
- 回表、filesort、临时表都会影响最终成本。
- 统计信息过旧会导致成本模型偏离真实执行情况。

## 项目经验版

线上遇到优化器选错索引时，我会先通过慢日志和 traceId 确认真实 SQL，再在只读环境对比 EXPLAIN 和执行耗时。紧急情况下可以灰度加 FORCE INDEX 止血；事后补充合适联合索引、刷新统计信息，并把该 SQL 加入慢 SQL 巡检。

## 常见追问

- FORCE INDEX 和 USE INDEX 有什么区别？
- 为什么统计信息会不准？
- 如何设计联合索引避免选错？
- EXPLAIN 里 rows 和 filtered 怎么看？

## 易错点

- 不要一上来就永久写死 FORCE INDEX。
- 不要只看 EXPLAIN，不看真实执行耗时。
- 不要忽略数据分布和参数值对执行计划的影响。
