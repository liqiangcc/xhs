# 如何解决高并发下的库存抢购超卖少买问题?

## 核心结论

库存抢购要把“扣减库存”设计成原子操作，并结合限流、预扣库存、异步下单、幂等和最终一致性。常见方案是 Redis Lua 预扣加消息队列削峰，数据库乐观锁做最终落库兜底。

## 1 分钟版

先在活动开始前把库存预热到 Redis。用户抢购时通过 Lua 脚本原子判断库存和用户资格，成功后扣 Redis 库存并写入下单消息。订单服务异步消费消息，使用数据库唯一键和乐观锁扣减真实库存，防止重复下单和超卖。如果后续创建订单失败，要通过补偿把库存回滚或释放。入口层还要做限流、验证码、排队和风控，减少无效流量。

## 3 分钟版

超卖通常来自并发读库存后分别扣减，解决关键是让库存判断和扣减不可拆分。Redis Lua 可以在单线程执行模型下完成库存校验、用户去重、扣减三个动作，适合作为高并发预扣层。数据库层不能完全信任 Redis，要用 `stock > 0` 的条件更新、版本号乐观锁或唯一约束保证最终一致。少买可能来自预扣后订单失败、消息丢失或库存未释放，因此需要订单超时关闭、库存补偿、消息可靠投递和对账任务。

## 关键细节

- Redis 扣减必须原子化，不能 get 后再 decr。
- 用户维度要做幂等，防止重复抢购。
- 数据库最终扣减要有条件更新或乐观锁。
- 异步链路要处理消息丢失、重复消费和补偿。

## 原理机制

- Lua 脚本把多步 Redis 操作合并为一次原子执行。
- MQ 削峰把瞬时流量转换成可控消费速率。
- 唯一键和幂等表防止重复创建订单。
- 对账任务修复 Redis、订单、库存之间的不一致。

## 项目经验版

我会把秒杀库存和普通库存隔离，活动前预热并冻结秒杀库存。抢购成功只代表获得下单资格，真实订单创建异步完成。所有关键链路记录活动 id、用户 id、库存流水和消息 id，方便补偿和对账。

## 常见追问

- Redis 扣成功但订单失败怎么办？
- 如何防止用户重复下单？
- 为什么还需要数据库乐观锁？
- 库存少买怎么发现和修复？

## 易错点

- 不要用数据库单点直接扛所有抢购流量。
- 不要把 Redis 预扣当成最终一致的全部保证。
- 不要忽略失败补偿和库存对账。
