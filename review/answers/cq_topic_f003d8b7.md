# 线上的API接口响应比较慢，该如何快速排查和定位问题？

## 核心结论

线上 API 慢要先确认影响范围和时间点，再沿调用链从入口、应用、下游、数据库、缓存、网络和机器资源逐层定位。核心方法是指标看趋势，trace 定位置，日志看原因，必要时用 Arthas 等工具现场诊断。

## 1 分钟版

先看监控：慢的是单接口还是全站，P99 还是平均值，是否伴随错误率上升。再查链路追踪，定位耗时在网关、应用逻辑、RPC、DB、Redis、MQ 还是外部接口。数据库看慢 SQL、锁等待、连接池；应用看线程池、GC、CPU、内存、日志异常；缓存看命中率和热 key。定位后先止血，比如限流、降级、扩容、回滚，再做根因修复。

## 3 分钟版

排查顺序要避免盲目猜。第一步确认变更和流量：是否刚发布、配置变更、活动流量、下游故障。第二步看黄金指标：QPS、RT、错误率、资源饱和度。第三步通过 trace 拆分每个 span 的耗时，如果数据库耗时高，就看执行计划、慢日志、锁和连接池；如果应用耗时高，就看线程堆栈、锁竞争、GC 和 CPU 火焰图；如果下游耗时高，就看超时、重试和熔断。最后给出短期止血和长期优化。

## 关键细节

- 先判断是局部接口慢还是整体系统慢。
- P95/P99 比平均耗时更能反映用户体验。
- trace 能快速定位慢在哪一段。
- 止血动作和根因修复要分开。

## 原理机制

- 调用链耗时等于入口处理、业务逻辑、依赖调用和排队等待之和。
- 线程池和连接池耗尽会造成排队延迟。
- 慢 SQL、锁等待和 GC 都会放大尾延迟。
- 重试风暴可能让下游慢变成全链路慢。

## 项目经验版

我会先打开该接口的 trace 样本和最近发布记录，确认慢调用集中在哪个依赖。如果是 SQL，先用索引或限流止血；如果是下游，先降级或熔断；如果是应用线程打满，用线程 dump 看阻塞点。处理过程中保持记录，方便复盘和加监控。

## 常见追问

- 没有链路追踪怎么排查？
- 如何判断是数据库慢还是应用慢？
- Arthas 可以看哪些信息？
- 接口偶发慢怎么定位？

## 易错点

- 不要一开始就猜数据库问题。
- 不要只看平均 RT。
- 不要让重试把故障进一步放大。
