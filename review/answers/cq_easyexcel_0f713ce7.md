# 场景：百万级别数据的Excel如何快速导入到数据库中

## 核心结论

百万 Excel 导入要避免一次性全量读入内存，采用流式解析、分批校验、批量写库、异步任务和可恢复进度。核心目标是控制内存、提升吞吐、保证幂等和可追踪。

## 1 分钟版

可以用 EasyExcel 这类流式解析工具逐行读取，按固定 batch 聚合，比如 500 到 2000 行一批。每批先做基础校验和转换，再批量写入数据库。导入过程放到后台任务中，前端只拿任务 id 查询进度。数据库写入要使用批量 insert 或批量 upsert，并设计唯一键保证重复导入不产生脏数据。失败行要单独记录，支持下载错误报告。

## 3 分钟版

整体流程可以分为上传、解析、校验、入库、结果反馈。上传后先生成导入任务，文件落对象存储或临时目录；解析端使用监听器流式读取，避免把整个 Excel 放进内存；校验分成格式校验、业务校验和唯一性校验；写库时按批提交，控制事务大小，避免长事务和锁等待。对于百万级数据，单机多线程要注意不要把数据库打满，可以按 sheet 或分片队列并行，但要有连接池、限流和失败重试控制。

## 关键细节

- 不要使用一次性读取整个 workbook 的方式。
- 批大小要结合数据库、字段数量和事务耗时压测调整。
- 需要导入任务表记录状态、进度、成功数、失败数。
- 幂等通常依赖业务唯一键、导入批次号或 upsert 逻辑。

## 原理机制

- 流式解析降低内存占用。
- 批量写库减少网络往返和 SQL 执行开销。
- 异步任务把长耗时操作从请求线程中剥离。
- 失败明细让导入过程可恢复、可审计。

## 项目经验版

项目里我会把导入做成任务化能力：上传后立即返回任务 id，后台消费者解析文件并分批入库。每批提交后更新进度，失败行写错误表。对核心表导入会先进入临时表，校验通过后再合并到正式表，降低对线上查询的影响。

## 常见追问

- EasyExcel 为什么比 POI 全量读取更适合大文件？
- 批量写入失败后如何处理？
- 如何避免重复导入？
- 多线程导入怎么控制数据库压力？

## 易错点

- 不要在 Web 请求线程里同步处理百万行。
- 不要用一个大事务覆盖全文件。
- 不要只追求并发数，数据库写入能力才是瓶颈之一。
