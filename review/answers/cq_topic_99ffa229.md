# MySQL索引失效的常见场景

## 核心结论

MySQL 索引失效通常来自查询条件无法利用有序索引结构，常见包括对索引列做函数或计算、类型不一致、like 前缀通配、违反最左前缀、范围条件后列无法继续有效利用、低选择性字段和优化器成本选择。

## 1 分钟版

常见场景有：where 中对索引列使用函数或表达式；字符串字段用数字比较导致隐式转换；like 以 `%` 开头；联合索引没有从最左列开始使用；范围查询后面的联合索引列无法继续用于精确定位；or 条件两边不是都有合适索引；使用 `!=`、`not in` 等导致扫描大量数据。最终是否走索引要看 EXPLAIN，而不是只看是否建了索引。

## 3 分钟版

索引本质是有序数据结构，查询条件要能转换成有序范围扫描才容易用上索引。比如 `date(create_time)=...` 会破坏列原始值的有序性，应该改成范围查询。联合索引遵循最左前缀原则，遇到范围条件后，后续列通常不能继续用于缩小扫描范围，但可能用于覆盖索引。优化器还会根据成本决定是否走索引，如果返回行比例太高，走全表扫描可能更便宜。

## 关键细节

- 对索引列做函数、计算可能失效。
- 隐式类型转换可能让索引列无法使用。
- `like '%abc'` 通常不能用普通 B+ 树索引。
- 联合索引要关注最左前缀和范围条件位置。

## 原理机制

- B+ 树索引依赖有序键值做范围定位。
- 条件如果不能形成连续范围，就难以高效扫描。
- 优化器会基于统计信息选择成本更低的计划。
- 覆盖索引可以减少回表，即使过滤能力有限也可能有价值。

## 项目经验版

排查慢 SQL 时我会先看 EXPLAIN 的 key、rows、Extra，再对照建表语句和真实参数。修复通常包括改写条件、统一字段类型、调整联合索引顺序、增加覆盖索引，最后通过压测和慢日志确认效果。

## 常见追问

- 最左前缀原则是什么？
- 范围查询后面的列还能用索引吗？
- 为什么隐式转换会导致索引失效？
- EXPLAIN 里怎么判断是否回表？

## 易错点

- 不要认为建了索引就一定会使用。
- 不要只背规则，要结合 B+ 树有序性解释。
- 不要忽略优化器可能主动选择全表扫描。
