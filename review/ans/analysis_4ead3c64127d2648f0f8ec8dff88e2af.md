## 🏗 Redis和MySQL的一致性应该如何保证

### 1. 需求拆解
**功能性需求**：
- 给出缓存与数据库一致性的标准回答框架。
- 区分缓存旁路、双写、异步订阅等方案的适用场景。
- 说明高并发下的不一致窗口、补偿手段、降级策略。

**非功能性需求**：
- 常见互联网业务读多写少，读写比例可能是 9:1 甚至 99:1。
- 核心指标是一致性窗口、数据库负载、缓存命中率、恢复速度。
- 方案要足够简单可落地，否则维护成本会超过收益。

### 2. 核心架构方案
标准答案优先从 Cache Aside 讲起：
- 读：先读 Redis，未命中再读 MySQL，并回填缓存。
- 写：先写 MySQL，成功后删除 Redis。
- 补偿：结合延时双删、MQ 重试、Binlog 订阅保证最终一致。

更完整的生产级演进：
1. 基础版：写 DB 后删缓存。
2. 加强版：删缓存失败进入重试队列。
3. 稳定版：订阅 Binlog，把“删除/刷新缓存”做成独立异步链路。
4. 高价值场景：引入版本号、逻辑时钟或 CDC 校对任务，防止旧值回写。

WHY：
- 一致性问题本质上来自两个存储系统的非原子双写。
- 既然无法低成本做到分布式强一致，就要让一个系统当真相源，另一个系统可重建、可补偿。

### 3. 核心链路详设
**写链路**：
- 应用层开启 MySQL 事务并更新数据。
- 事务提交成功后删除对应 Redis Key。
- 删除失败时记录日志、重试或投递 MQ。
- 对于关键业务，Binlog 订阅程序消费变更，再次校准缓存。

**读链路**：
- Redis 命中：直接返回，RT 约 1 ms。
- Redis 未命中：查询 MySQL，RT 约 5~20 ms，随后回填缓存并设置 TTL。
- 热点 Key 采用互斥构建，避免缓存击穿。

一套面试上好用的回答顺序：
- 先承认“绝对一致难且没必要泛化”。
- 再给主流方案“DB 为准 + 删除缓存”。
- 最后补充“高并发下靠异步补偿与版本校验收敛”。

### 4. 关键技术选型 & 理由
| 组件 | 选型 | 理由 | 备选方案 |
|------|------|------|----------|
| 主存储 | MySQL | 强事务、约束强、可持久恢复 | Redis 作为真相源不稳妥 |
| 缓存模式 | Cache Aside | 业务侵入低、工程成熟 | Write Through 需要更重中间层 |
| 修正机制 | MQ / Binlog CDC | 自动补偿，减少人工介入 | 仅靠定时任务修复不够实时 |
| 热点控制 | 互斥重建 + TTL + 随机过期 | 兼顾一致性与抗击穿 | 单纯永久缓存不可控 |

### 5. 异常与容灾
| 故障场景 | 影响 | 应对策略 |
|----------|------|----------|
| 删除缓存失败 | 用户短时间读到旧值 | 重试、MQ、Binlog 补偿 |
| 缓存击穿 | DB 瞬时压力暴涨 | 热点互斥、逻辑过期、限流 |
| 回源读从库延迟 | 回填旧值污染缓存 | 关键链路读主库，或使用版本校验 |
| Redis 故障 | 命中率骤降，DB 被打爆 | 熔断、限流、降级、缓存预热 |

### 6. 性能估算（Back-of-the-Envelope）
- 假设日活接口读写比 95:5、总 QPS 为 1 万，缓存命中率 98% 时，MySQL 读压约 200 QPS；若 Redis 故障导致命中率跌到 50%，MySQL 压力会瞬间涨到 5000 QPS。
- 如果缓存删除失败率为 $10^{-4}$，每天 2000 万次写就会留下约 2000 个脏 Key，因此必须设计自动补偿，而不是靠人工。
- 延时双删第二次删除如果设在 500 ms~2 s，通常能覆盖大多数“旧读回填”窗口，但它只是降低概率，不是严格证明的一致性方案。

### 7. 面试官追问预判 🔮

- 追问 2：Binlog + Canal 修复缓存的一致性链路，如何保证消息不丢和幂等？
- 追问 3：如果业务是账户余额或库存扣减，这套最终一致方案还能直接用吗？
