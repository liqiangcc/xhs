## 🏗 讲讲 Redis 和 Mysql 保持数据一致性的方式？

### 1. 需求拆解
**功能性需求**：
- 解释 Redis 与 MySQL 双写场景里为什么会不一致。
- 给出工程上主流的一致性方案，并说明适用边界。
- 能落到缓存读写、失效、回填、重试、补偿这些具体动作。

**非功能性需求**：
- 典型缓存场景要求读延迟 1~5 ms，数据库写延迟 5~20 ms。
- 一致性目标通常不是“绝对强一致”，而是“最终一致 + 窗口可控”。
- 系统需要兼顾吞吐、复杂度、可运维性，不能为了极致一致性牺牲整体可用性。

### 2. 核心架构方案
最常见、最稳妥的方案不是“先写 Redis 再写 MySQL”，而是：
1. 写请求只以 MySQL 为准进行事务提交。
2. 提交成功后删除缓存，而不是直接更新缓存。
3. 读请求采用 Cache Aside：先查 Redis，没有命中再查 MySQL，并把结果回填 Redis。
4. 对高并发热点数据，再叠加延时双删、消息重试或 Canal 订阅 Binlog 做异步修正。

WHY：
- MySQL 是持久化真相源，事务能力强；Redis 更像性能层。
- “更新缓存”会引入更多并发覆盖问题，而“删除缓存”把复杂度转移到了下一次读回源。
- 真正难点不是“有没有不一致”，而是“不一致窗口有多大，能不能自动收敛”。

### 3. 核心链路详设
**写链路**：
- 方案 A（推荐）：更新 MySQL → 提交事务 → 删除 Redis Key。
- 方案 B（加强版）：更新 MySQL → 删除 Redis → 发送 MQ/延迟任务再次删一次 Redis。
- 方案 C（重流量/多系统）：更新 MySQL → Binlog → Canal/Flink → 异步刷新或删除 Redis。

**读链路**：
- 先查 Redis，命中则直接返回。
- 未命中则查 MySQL，把结果写回 Redis，并设置 TTL。
- 为避免击穿，可对热点 Key 加互斥重建或逻辑过期。

关键失败点：
- MySQL 提交成功但缓存删除失败。
- 删除缓存后，旧请求又把脏数据回填到 Redis。
- 主从延迟导致“读从库 + 回填缓存”写入旧值。

### 4. 关键技术选型 & 理由
| 组件 | 选型 | 理由 | 备选方案 |
|------|------|------|----------|
| 一致性模式 | Cache Aside | 简单、成熟、符合大多数读多写少场景 | Read/Write Through 对中间层要求更高 |
| 写策略 | 更新 DB 后删缓存 | 避免双写更新覆盖，失败面更小 | 先更新缓存再更新 DB 风险更高 |
| 补偿机制 | 延时双删 / MQ 重试 / Binlog 订阅 | 用异步机制缩短并修复不一致窗口 | 人工修复不可持续 |
| 数据源 | MySQL 为真相源 | 事务、持久化、约束能力更强 | Redis 不适合承担最终事实库角色 |

### 5. 异常与容灾
| 故障场景 | 影响 | 应对策略 |
|----------|------|----------|
| DB 更新成功，删除缓存失败 | Redis 残留旧值 | 重试删除、发送 MQ、订阅 Binlog 补偿 |
| 删除缓存后旧读请求回填脏数据 | 缓存被污染 | 回填前校验版本号/更新时间，或使用延时双删 |
| 主从延迟 | 回源读到旧数据 | 关键写后读走主库，或引入短期读一致性策略 |
| 热点 Key 失效 | 大量请求打到 DB | 互斥锁、逻辑过期、预热 |

### 6. 性能估算（Back-of-the-Envelope）
- 假设某热点数据 QPS 为 5000，缓存命中率 95%，则只有约 250 QPS 打到 MySQL；如果缓存一致性失控，命中率降到 60%，MySQL 压力会变成 2000 QPS，可能直接压垮主库。
- 一次删除缓存通常 < 1 ms，而一次 MySQL 提交 5~20 ms，因此以 DB 为准、缓存异步收敛是更现实的取舍。
- 如果删除失败率是万分之一，单日 1 亿次写请求也会产生约 1 万次潜在脏缓存，所以必须配套重试或 Binlog 补偿。

### 7. 面试官追问预判 🔮
- 追问 1：为什么“更新数据库后删除缓存”比“更新数据库后更新缓存”更稳？
- 追问 2：延时双删到底解决了什么问题，为什么不能保证绝对一致？
- 追问 3：如果业务要求库存场景强一致，Redis 和 MySQL 之间还能继续用普通缓存方案吗？
