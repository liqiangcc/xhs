## 🔬 Redis 的事务。如何理解 Mysql 和 Redis 的事务？能举出应用的例子吗？

### 1. 问题本质
> 这道题的核心不是比较两个“都叫事务”的功能，而是看你是否理解：MySQL 事务解决的是数据库并发读写下一致性与持久性问题，而 Redis `MULTI/EXEC` 更像“命令批量串行执行 + 有限隔离”，两者语义根本不同。

### 2. 全景架构图
可以把两者理解成两种完全不同的执行模型：
- MySQL/InnoDB：客户端发起事务 → 存储引擎通过 Undo Log、Redo Log、锁、MVCC 协调并发 → `COMMIT` 时落盘并保证 ACID 中的大部分语义。
- Redis：客户端发送 `MULTI` 后命令进入队列 → `EXEC` 时由单线程顺序执行 → 如配合 `WATCH`，只是在提交前做乐观冲突检查。

所以 MySQL 事务是“并发控制系统的一部分”，Redis 事务是“原子批处理接口的一部分”。

### 3. 执行流程深度拆解
**Step 1: 开启事务的方式不同**
- MySQL：`BEGIN`/`START TRANSACTION` 后，事务获得一致性视图，并可能在后续语句中加锁。
- Redis：`MULTI` 后并不真正执行命令，只是把命令入队。
- WHY：MySQL 要处理复杂并发；Redis 单线程执行命令，天然避免单命令级别的并发竞争。

**Step 2: 提交前的控制机制不同**
- MySQL：通过锁和 MVCC 保证隔离级别；写操作会写 Undo/Redo，必要时阻塞其他事务。
- Redis：`WATCH key` 只是 CAS 风格的乐观校验，若被改过，`EXEC` 直接失败返回空结果。
- WHY：Redis 不想引入重型锁管理，否则会破坏其高吞吐与简单模型。

**Step 3: 失败语义不同**
- MySQL：事务中某条 SQL 失败，可根据异常决定回滚，`ROLLBACK` 后数据回到一致状态。
- Redis：命令排队阶段语法错误会导致整个事务失败；执行阶段某条命令报错，其他命令仍会继续执行，不会像 MySQL 一样自动回滚。
- WHY：Redis 事务不维护命令级 undo 信息，回滚成本和设计复杂度太高。

**Step 4: 持久性保障不同**
- MySQL：依赖 Redo Log + Binlog + `fsync` 策略，可做到崩溃恢复。
- Redis：是否持久取决于 RDB/AOF 配置；事务本身不等于强持久。
- WHY：Redis 首先是内存数据库，事务语义不能脱离其持久化模式单独谈。

**Step 5: 正确使用方式不同**
- MySQL 事务适合订单、转账、库存扣减等强一致写场景。
- Redis 事务适合计数累加、批量写入、乐观锁更新少量 Key；更复杂原子逻辑通常直接用 Lua 脚本。
- WHY：Lua 脚本在 Redis 中往往比 `MULTI/EXEC` 更常用，因为能把判断与写入放在一次原子执行里。

### 4. 关键设计决策 & Tradeoff
| 设计决策 | 选择了什么 | 放弃了什么 | 为什么 |
|----------|-----------|-----------|--------|
| MySQL 并发控制 | 锁 + MVCC + 日志 | 极致简单执行模型 | 需要 ACID 与复杂查询能力 |
| Redis 事务模型 | 队列化命令 + 单线程串行 | 完整回滚与强隔离 | 保持高性能和实现简洁 |
| Redis 冲突处理 | `WATCH` 乐观锁 | 悲观锁长时间持有 | 大多数 Key 操作短平快，冲突可重试 |
| 复杂原子逻辑 | Lua 脚本 | 多命令间复杂状态回滚 | 脚本一次执行更贴近 Redis 能力边界 |

### 5. 极端场景分析
- 高并发下会怎样？
  - MySQL 可能出现锁等待、死锁、回滚重试。
  - Redis 不会有单命令并发冲突，但热点 Key 上 `WATCH` 失败率可能升高，客户端需要不断重试。
- 数据量极大时会怎样？
  - MySQL 事务太大，会导致 Undo 膨胀、持锁时间长、主从延迟上升。
  - Redis 若在事务或 Lua 中操作大量 Key，会阻塞主线程，放大请求尾延迟。
- 出错/异常时会怎样？
  - MySQL 宕机后可以依赖日志恢复到提交点。
  - Redis 若未开启合适 AOF 策略，即使事务执行成功，也可能在宕机后丢失最近数据。

- 追问 3：如果要做库存扣减，什么时候应该放在 MySQL 事务里做，什么时候可以放在 Redis 原子操作里做？
