## 🔬 mysql的acid特性是怎么实现的

### 1. 问题本质
> ACID 不是一个单点特性，而是 MySQL/InnoDB 用“日志系统 + 锁/MVCC + 崩溃恢复机制”共同实现的结果：既要让事务看起来原子一致，又要在高并发和宕机后仍然成立。

### 2. 全景架构图
可以把 InnoDB 的事务系统看成四层协作：
- 执行层：SQL 在事务里被解析和执行。
- 并发控制层：锁、MVCC、Read View 控制不同事务互相可见什么。
- 恢复层：Undo Log 负责回滚旧版本，Redo Log 负责崩溃后重做已提交修改。
- 外部一致性层：Binlog 负责主从复制和归档，配合两阶段提交保证 Redo 与 Binlog 一致。

ACID 里的每个字母都不是一个组件单独实现的，而是多个机制拼起来的。

### 3. 执行流程深度拆解
**Step 1: Atomicity（原子性）如何实现**
- 事务修改数据前，会记录逻辑旧值到 Undo Log。
- 如果事务执行失败或显式 `ROLLBACK`，系统根据 Undo Log 逆向恢复。
- WHY：数据库不可能每次出错都“重新拷贝一份整表”，必须以日志方式低成本回退。

**Step 2: Consistency（一致性）如何实现**
- 一致性不是单一模块，而是依赖事务原子性、约束、触发器、业务规则共同保证。
- 数据库层负责主键、唯一键、外键、隔离控制等底线；业务层负责更高层语义。
- WHY：数据库可以保证“不会写坏”，但不能单独理解“余额不能为负”这类完整业务含义。

**Step 3: Isolation（隔离性）如何实现**
- 快照读靠 MVCC：每行有隐藏版本字段，事务通过 Read View 决定能看到哪个版本。
- 当前读靠锁：`for update`、`update`、`delete` 会加行锁、间隙锁、Next-Key Lock。
- WHY：只靠锁并发太差，只靠 MVCC 又挡不住所有写冲突，所以两者混用。

**Step 4: Durability（持久性）如何实现**
- 数据页先在 Buffer Pool 里改，不会每次都直接刷磁盘。
- 提交时先刷 Redo Log；宕机恢复时依赖 Redo 把已提交但未落盘的数据页重做出来。
- `innodb_flush_log_at_trx_commit=1` 时，每次提交都尽量刷盘，持久性最强。
- WHY：WAL（Write-Ahead Logging）比“先刷数据页再提交”高效得多。

**Step 5: 为什么还需要 Binlog 和两阶段提交**
- Redo Log 是 InnoDB 内部物理日志，Binlog 是 Server 层逻辑日志。
- 若两者写入不同步，主从复制或恢复会出现“主库提交了，从库没看到”的问题。
- 两阶段提交大致流程：`prepare redo` → `write binlog` → `commit redo`。
- WHY：这样能把存储引擎内部恢复能力和数据库外部复制能力对齐。

### 4. 关键设计决策 & Tradeoff
| 设计决策 | 选择了什么 | 放弃了什么 | 为什么 |
|----------|-----------|-----------|--------|
| 回滚机制 | Undo Log | 直接拷贝旧页快照 | 日志开销更可控，粒度更细 |
| 持久化机制 | Redo Log + WAL | 每次提交直接刷数据页 | 刷顺序日志远快于随机刷脏页 |
| 并发控制 | MVCC + 锁混合 | 纯锁 / 纯版本控制 | 在一致性与性能之间折中 |
| 复制一致性 | Binlog + 两阶段提交 | Redo/Binlog 各写各的 | 避免主从与恢复状态不一致 |

### 5. 极端场景分析
- 高并发下会怎样？
  - 锁竞争会升高，Undo 链变长，长事务还会拖慢 purge，导致历史版本清理压力上升。
- 数据量极大时会怎样？
  - 脏页刷盘、checkpoint、Redo 生成速率成为瓶颈；不是 ACID 失效，而是成本上升。
- 出错/异常时会怎样？
  - 宕机后依据 Redo 重做，未提交事务依赖 Undo 回滚；若配置较弱的刷盘策略，则最近提交的数据可能存在丢失窗口。
