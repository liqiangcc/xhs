## 🎯 技术选型：搜索场景下：为什么模糊查询推荐使用 Elasticsearch (ES) 而非 MySQL？ES 检索更快的底层原理为何？

### 1. 一句话定义
> 搜索场景优先选 Elasticsearch，不是因为 MySQL 不能查，而是因为 ES 基于倒排索引为“关键词检索、相关性排序、海量文本召回”而生，而 MySQL 的 B+ 树索引更擅长精确匹配、范围查询和事务型读写。

### 2. 为什么需要它？（解决了什么问题）
- `like '%keyword%'` 在 MySQL 中通常无法高效利用普通索引，容易退化为全表扫描，数据量一大就慢。
- 搜索系统除了“查到”，还要“查得准、查得快、能排序、能分词、能模糊匹配”。
- ES 的设计目标就是把全文拆词后建立倒排索引，让查询复杂度从“扫文档”变成“查词项 posting list”。

### 3. 核心知识点拆解
| 维度 | 要点 | 关键细节 |
|------|------|----------|
| MySQL 索引模型 | B+ 树 | 擅长 `=`、`BETWEEN`、前缀匹配；不擅长任意位置模糊检索 |
| ES 索引模型 | 倒排索引 | term → doc list，先找词，再找文档 |
| 文本处理 | Analyzer 分词 | 会做分词、归一化、停用词过滤 |
| 排序能力 | BM25/TF-IDF 思路 | 不只是返回结果，还能按相关性排序 |
| 横向扩展 | 分片 + 副本 | 面向搜索流量扩展更自然 |
| 更新代价 | ES 写入较重 | 需要 refresh/segment merge，实时性通常是近实时 |

### 4. 横向对比（如有近似概念）
| 对比项 | MySQL | Elasticsearch |
|--------|---|---|
| 适用场景 | 事务型系统、精确查询、关系约束 | 全文检索、复杂召回、搜索推荐 |
| 优势 | ACID 强、JOIN 能力强、数据一致性好 | 模糊搜索强、相关性排序强、海量检索快 |
| 劣势 | `%xx%` 模糊检索性能差 | 强事务弱、写链路复杂、需要额外运维 |

| 对比项 | `LIKE 'abc%'` | `LIKE '%abc%'` |
|--------|---|---|
| 索引利用 | 可能走前缀索引 | 通常无法有效利用 B+ 树 |
| 性能 | 相对可控 | 容易全表扫描 |

### 5. 面试官追问预判 🔮
- 追问 1：倒排索引和 B+ 树索引的本质差异是什么，为什么一个适合搜索、一个适合数据库？
- 追问 2：ES 为什么是“近实时”而不是严格实时？`refresh` 和 segment merge 会带来什么影响？
- 追问 3：如果业务既要事务强一致，又要全文搜索，MySQL 与 ES 的数据同步链路应该怎么设计？

### 6. 记忆锚点 🧠
> 记忆类比：MySQL 像“按目录翻书”，ES 像“先建好全书关键词索引卡片”。查精确章节用目录，查散落全文关键词用卡片索引。
