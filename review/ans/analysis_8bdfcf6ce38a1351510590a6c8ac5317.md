## 🏗 场景：给定一个sql语句，询问加了什么锁？

### 1. 需求拆解
**功能性需求**：
- 判断一条 SQL 在 InnoDB 下会拿到什么锁：表锁、意向锁、Record Lock、Gap Lock、Next-Key Lock。
- 结合事务隔离级别、索引命中情况、等值/范围条件，解释为什么会加这类锁。
- 能说明是否会阻塞其他事务，以及阻塞范围。

**非功能性需求**：
- 面试里通常默认 OLTP 场景，单库 QPS 在 1k~10k。
- 锁判断必须关注延迟与并发，错误判断会直接导致死锁、锁等待、吞吐下降。
- 要能快速定位：语句类型、执行计划、隔离级别、索引条件。

### 2. 核心架构方案
锁判断的核心不是背锁名，而是沿着一条链路推：
1. 先确认是不是 `SELECT`、`SELECT ... FOR UPDATE/LOCK IN SHARE MODE`、`UPDATE`、`DELETE`、`INSERT`。
2. 再看隔离级别，MySQL/InnoDB 在 `RR` 下更容易引入 Gap/Next-Key Lock 防幻读，在 `RC` 下 Gap Lock 明显减少。
3. 再看是否走索引以及条件是否唯一命中：唯一索引等值命中通常退化为行锁；范围查询或未命中索引时，锁范围会扩大。
4. 最后看执行计划。如果 SQL 实际走了全表扫描，面试里经常意味着“你以为锁一行，实际上锁了一大片扫描范围”。

### 3. 核心链路详设
**写链路**：请求入口 → 解析 SQL 类型 → 优化器选择索引 → 存储引擎按索引项扫描 → 在扫描到的索引记录/间隙上加锁

**读链路**：
- 普通 `SELECT`：快照读，依赖 MVCC，一般不加锁。
- `SELECT ... FOR UPDATE`：当前读，会加排他锁。
- `SELECT ... LOCK IN SHARE MODE`（或新语法 `FOR SHARE`）：当前读，会加共享锁。

典型判断规则：
- `UPDATE user SET ... WHERE id = 10` 且 `id` 是主键：Record Lock 为主，同时表级有意向排他锁 `IX`。
- `UPDATE user SET ... WHERE age = 20` 且 `age` 非唯一索引：常见是 Next-Key Lock，锁住命中的索引记录及相邻间隙。
- `DELETE ... WHERE age > 20`：范围删除，通常锁的是一段索引区间，不只是现有记录。
- `INSERT INTO ... VALUES (...)`：核心是 Insert Intention Lock，若命中唯一索引冲突还会等待已有记录锁。
- 条件未走索引：可能退化为大量行扫描，锁冲突面急剧扩大。

### 4. 关键技术选型 & 理由
| 组件 | 选型 | 理由 | 备选方案 |
|------|------|------|----------|
| 存储引擎 | InnoDB | 支持行锁、MVCC、Next-Key Lock，才有面试里这些细粒度行为 | MyISAM 主要是表锁，不适合高并发更新 |
| 锁粒度判断 | 基于索引访问路径判断 | InnoDB 的行锁本质上是“锁索引项”，不是锁逻辑行 | 只按 SQL 文本猜测，容易误判 |
| 隔离策略 | `RR` 默认分析 | MySQL 默认 `RR`，面试官也通常以默认值追问 | `RC` 下部分 Gap Lock 行为不同 |
| 排查工具 | `EXPLAIN` + `SHOW ENGINE INNODB STATUS` + `performance_schema.data_locks` | 能把“加了什么锁”从猜测变成证据 | 仅靠业务日志无法定位锁细节 |

### 5. 异常与容灾
| 故障场景 | 影响 | 应对策略 |
|----------|------|----------|
| 未命中索引导致扫描范围过大 | 锁等待时间飙升，吞吐下降 | 补合适索引，确保 where 条件可命中 |
| 范围更新引发 Gap/Next-Key Lock | 并发插入/更新被阻塞 | 缩小范围、拆批、改成更精确索引条件 |
| 事务过长 | 持锁时间长，放大死锁概率 | 控制事务长度，尽早提交 |
| 死锁 | 一个事务回滚，业务报错 | 固定访问顺序、缩短事务、开启重试 |

### 6. 性能估算（Back-of-the-Envelope）
- 假设单条 `UPDATE ... WHERE id=?` 命中主键，锁 1 条记录，持锁 5 ms，则理论单热点行每秒最多约 $1000/5=200$ 次串行更新。
- 若范围更新一次扫描并锁定 1 万条索引项，事务持续 200 ms，那么同范围内其他写请求几乎都会阻塞，尾延迟会从毫秒级放大到百毫秒甚至秒级。
- 如果一个无索引条件触发全表扫描，10 万行表在高峰期会形成“锁放大器”，TPS 可能直接腰斩。

### 7. 面试官追问预判 🔮
- 追问 1：为什么 InnoDB 的“行锁”本质上是锁索引，而不是锁数据行本身？
- 追问 2：`RR` 下如何通过 Next-Key Lock 解决幻读？在 `RC` 下为什么行为不同？
- 追问 3：如果同一条 SQL 在你预期里只锁 1 行，但线上却出现大量锁等待，你会如何排查执行计划和锁信息？
