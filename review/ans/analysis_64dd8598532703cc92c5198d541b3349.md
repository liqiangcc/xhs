## 🎯 技术选型：场景化分析：什么场景下优先使用 MySQL？什么场景下必须引入 Redis？

### 1. 一句话定义
> MySQL 适合作为“真相源”和事务型存储，Redis 适合作为“高性能访问层”和内存型数据结构服务；是否必须引入 Redis，取决于你的读写延迟目标、并发峰值和数据访问模式是否已经超出单靠 MySQL 的舒适区。

### 2. 为什么需要它？（解决了什么问题）
- 只靠 MySQL 能把数据存下来，但在热点读、高并发计数、分布式锁、排行榜等场景下延迟和吞吐会不够。
- 只靠 Redis 性能很强，但事务约束、复杂查询、持久化真相和关系模型都不足。
- 工程上不是二选一，而是职责分层：MySQL 管正确，Redis 管快。

### 3. 核心知识点拆解
| 维度 | 要点 | 关键细节 |
|------|------|----------|
| MySQL 优势 | ACID、关系模型、索引、JOIN | 适合订单、账户、库存、用户资料等核心数据 |
| Redis 优势 | 内存访问、丰富数据结构、亚毫秒级响应 | 适合缓存、计数器、排行榜、会话、限流 |
| 何时优先 MySQL | 写多于读、要求强一致、需要复杂查询 | 如支付、账务、配置主数据 |
| 何时必须引入 Redis | 热点读 QPS 高、延迟要求严、需要原子数据结构操作 | 如秒杀库存预扣、Feed 计数、分布式锁 |
| 常见组合方式 | MySQL + Redis Cache Aside | DB 为准，Redis 做性能层 |
| 风险点 | 双写一致性、缓存穿透/击穿/雪崩 | 需要 TTL、限流、补偿机制 |

### 4. 横向对比（如有近似概念）
| 对比项 | MySQL | Redis |
|--------|---|---|
| 适用场景 | 核心业务主存储 | 热点访问与高性能中间层 |
| 优势 | 强一致、持久化稳定、查询能力强 | 低延迟、高吞吐、数据结构丰富 |
| 劣势 | 热点高并发下容易成为瓶颈 | 内存成本高，一致性与复杂查询较弱 |

| 对比项 | 只用 MySQL | MySQL + Redis |
|--------|---|---|
| 架构复杂度 | 低 | 更高 |
| 性能上限 | 受数据库瓶颈限制明显 | 能显著提升读吞吐和热点承载能力 |
| 适用阶段 | 早期业务或低并发系统 | 中高并发、热点明显的成熟系统 |

### 5. 面试官追问预判 🔮
- 追问 1：如果读写比是 100:1、热点 Key 极少但请求极集中，你会怎样设计 MySQL 与 Redis 的职责边界？
- 追问 2：既然 Redis 也能持久化，为什么它通常不能直接替代 MySQL 做核心真相库？
- 追问 3：引入 Redis 后最容易踩的坑是什么，怎么处理缓存一致性和雪崩问题？
