## 🎯 MySQL 事务的隔离级别，默认隔离级别是什么？

### 1. 一句话定义
> MySQL 事务隔离级别定义的是“一个事务在并发环境下能看到其他事务哪些中间状态”，默认隔离级别是 `Repeatable Read`（可重复读）。

### 2. 为什么需要它？（解决了什么问题）
- 没有隔离级别时，多事务并发会出现脏读、不可重复读、幻读，业务会读到“半成品数据”。
- 隔离越强，一致性越好，但锁冲突、等待时间和吞吐损失往往越大。
- 数据库设计的核心矛盾是：既想让并发高，又想让结果稳定；隔离级别就是这组取舍的显式开关。

### 3. 核心知识点拆解
| 维度 | 要点 | 关键细节 |
|------|------|----------|
| `Read Uncommitted` | 允许读未提交 | 可能发生脏读，几乎很少在生产使用 |
| `Read Committed` | 只能读到已提交数据 | 解决脏读，但可能不可重复读；Oracle 默认是它 |
| `Repeatable Read` | 同一事务内多次读取结果一致 | MySQL 默认级别；结合 MVCC 与 Next-Key Lock，大多数场景下也避免幻读 |
| `Serializable` | 最强隔离 | 通过更严格加锁近似串行执行，吞吐最低 |
| 默认值 | `RR` | InnoDB 默认不是拍脑袋选择，而是平衡了一致性与性能 |
| 实现机制 | MVCC + 锁 | 快照读主要靠 MVCC；当前读和范围保护还要靠锁 |

### 4. 横向对比（如有近似概念）
| 对比项 | `Read Committed` | `Repeatable Read` |
|--------|---|---|
| 适用场景 | 更偏 OLTP 通用并发写场景 | MySQL 默认业务场景，强调读取稳定性 |
| 优势 | 锁冲突相对少，读到最新提交结果 | 同事务多次查询结果一致，面试语境更常见 |
| 劣势 | 可能不可重复读，部分范围问题更明显 | 在范围更新/查询下锁行为更复杂 |

| 对比项 | `Repeatable Read` | `Serializable` |
|--------|---|---|
| 适用场景 | 大多数线上业务 | 极端要求严格串行一致的少数场景 |
| 优势 | 性能和一致性平衡较好 | 一致性最强 |
| 劣势 | 仍需理解幻读与当前读细节 | 并发性能最差，阻塞明显 |

### 5. 面试官追问预判 🔮
- 追问 1：MySQL 的 `RR` 为什么还能基本解决幻读？MVCC 和 Next-Key Lock 分别负责什么？
- 追问 2：为什么 Oracle 默认选 `RC`，而 MySQL/InnoDB 默认选 `RR`？
- 追问 3：快照读和当前读分别对应哪些 SQL，隔离级别对它们的影响有什么不同？

### 6. 记忆锚点 🧠
> 记忆口诀：`RU` 脏、`RC` 新、`RR` 稳、`SER` 串。MySQL 默认选“稳”而不是“最新”。
