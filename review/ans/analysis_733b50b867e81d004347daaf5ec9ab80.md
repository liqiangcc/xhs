## 🎯 MongoDB 和 MySQL 的区别是什么? 各自有哪些优势?

### 1. 一句话定义
> MySQL 是以关系模型、事务和结构化查询为核心的关系型数据库；MongoDB 是以文档模型、灵活 Schema 和水平扩展友好性为核心的文档型 NoSQL 数据库。

### 2. 为什么需要它？（解决了什么问题）
- 不同业务的数据形态差异极大：有的强调强一致和复杂关系，有的强调灵活字段和快速迭代。
- 关系型数据库擅长“结构稳定、约束明确、事务强”的系统；文档数据库擅长“结构变化快、嵌套对象多、扩展方便”的系统。
- 这类题的关键不是背特性，而是说明：为什么一种数据模型在某类业务下更自然、更省工程成本。

### 3. 核心知识点拆解
| 维度 | 要点 | 关键细节 |
|------|------|----------|
| 数据模型 | MySQL 是表/行/列；MongoDB 是 BSON 文档 | MongoDB 更适合嵌套结构 |
| Schema | MySQL 通常强 Schema | MongoDB 更灵活，字段可演进 |
| 查询能力 | MySQL SQL、JOIN、聚合成熟 | MongoDB 文档查询方便，但跨集合关系表达弱于关系库 |
| 事务能力 | MySQL 事务成熟、默认主力 | MongoDB 也支持事务，但常见心智仍偏文档操作 |
| 扩展方式 | MySQL 常靠分库分表、主从复制 | MongoDB 分片模型更原生 |
| 典型场景 | 订单、支付、账户、ERP | 内容中心、用户画像、配置中心、日志类文档数据 |

### 4. 横向对比（如有近似概念）
| 对比项 | MySQL | MongoDB |
|--------|---|---|
| 适用场景 | 强事务、复杂关系、报表与精确查询 | 文档结构灵活、字段多变、快速迭代 |
| 优势 | ACID 强、生态成熟、SQL 能力强 | Schema 灵活、嵌套对象友好、分片扩展便利 |
| 劣势 | 字段变化频繁时演进成本高 | 复杂关系查询和强约束能力较弱 |

| 对比项 | 规范化设计 | 文档聚合设计 |
|--------|---|---|
| 数据组织 | 拆表建关系 | 一份文档聚合更多上下文 |
| 优势 | 冗余少、约束强 | 读路径短、业务对象更完整 |
| 劣势 | JOIN 成本高 | 冗余与一致性维护成本上升 |

### 5. 面试官追问预判 🔮
- 追问 1：如果用户资料字段经常变化、还包含多层嵌套数组，你为什么可能更偏向 MongoDB？
- 追问 2：为什么订单、余额、库存这类系统通常仍优先选择 MySQL，而不是 MongoDB？
- 追问 3：在一个系统里同时用 MySQL 和 MongoDB 时，如何划分两者的边界，避免数据职责混乱？

### 6. 记忆锚点 🧠
> 记忆类比：MySQL 像“标准化表格档案室”，MongoDB 像“按对象打包的资料袋”。结构稳定选表格，结构多变选资料袋。
