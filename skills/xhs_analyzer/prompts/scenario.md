# 🏗 场景设计_Scenario 分析框架

> 适用：系统设计、方案设计、高并发/高可用场景分析
> 典型题干：「如何设计 XXX 系统」「XXX 场景下如何保证 YYY」「场景：...」

## 输出结构

```markdown
## 🏗 [原始题目]

### 1. 需求拆解
**功能性需求**：
- ...

**非功能性需求**：
- 预估 QPS / 数据量 / 并发数
- 延迟要求 / 可用性要求

### 2. 核心架构方案
描述整体架构设计，包含核心组件和数据流。

### 3. 核心链路详设
**写链路**：请求入口 → 核心处理 → 数据落地
**读链路**：缓存策略 → 降级策略

### 4. 关键技术选型 & 理由
| 组件 | 选型 | 理由 | 备选方案 |
|------|------|------|----------|

### 5. 异常与容灾
| 故障场景 | 影响 | 应对策略 |
|----------|------|----------|
| 缓存击穿 | | |
| 数据库宕机 | | |
| 消息积压 | | |

### 6. 性能估算（Back-of-the-Envelope）
- 存储量 = ...
- 带宽 = ...
- 单机 QPS = ...

### 7. 面试官追问预判 🔮
- 追问 1：如果流量翻 10 倍呢？
- 追问 2：如何保证数据一致性？
- 追问 3：如何做灰度发布/回滚？
```

## 分析要求

- **量化**：所有估算必须给出具体数字和计算过程
- **取舍透明**：每个选型都要说清"为什么不选另一个"
- **故障驱动**：从"哪里会出问题"倒推设计
- **分层递进**：先给最简方案，再逐步演进到生产级

## 资深候选人口径补充

这个框架本身已经比较偏工程设计，但如果用户是 5 年/7 年以上候选人，还应额外强调：

- 先讲业务目标，再讲架构，不要一上来堆组件名
- 每个方案都补一句“代价是什么”
- 必须至少给一个真实线上问题：流量、稳定性、成本、数据一致性、回滚
- 对核心链路优先输出：
	- `结论`
	- `最小可行方案`
	- `生产级增强`
	- `故障与止血`

### 推荐附加输出

```markdown
### 7年经验版怎么答
- 结论：我会先满足业务目标，再按流量和稳定性做分层设计
- 原理：核心链路怎么跑
- 取舍：为什么选这个组件/模式，不选另一个
- 实践：线上如何扩容、监控、止血、回滚
- 边界：最容易出问题的地方是什么
```
