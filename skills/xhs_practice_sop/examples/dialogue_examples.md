请先按这个格式回答：一句话结论 + 3~5 个要点 + 每点一句 why/场景 + 一个风险/边界。

### 用户
结论：联合索引适合多个条件经常一起查询的场景。
1. 要先看真实高频 SQL，而不是想当然建索引。
2. 等值查询列一般放前面，范围查询列放后面，因为范围后面利用率会下降。
3. 如果有 order by，要把排序列一起考虑，不然会 filesort。
4. 还要考虑覆盖索引和回表成本。
风险：索引太多会影响写入性能。

使用 xhs_practice_sop 模拟美团社招后端一面

第 1 题：缓存击穿、穿透、雪崩分别是什么？线上如何系统性治理？
请先按点回答。

### 用户
结论：三者本质都是缓存失效后流量打到 DB，但成因不同。
1. 穿透是查不存在的数据。
2. 击穿是热点 key 失效。
3. 雪崩是大量 key 同时失效。
4. 解决上可以用布隆过滤器、互斥锁、随机过期时间和多级缓存。
风险：方案太多会增加复杂度。

追问 1：如果热点 key 失效时 QPS 很高，为什么只加互斥锁还不一定够？

### 用户
